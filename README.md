# Kaggle 道路事故风险预测：把物理直觉写进特征工程

![Result Card](assets/result_card.png)

## 这个项目在简历里的定位

这个项目虽然还没达到 top 20%，但很适合展示可解释特征工程。它不是简单把字段丢给模型，而是把速度、曲率、车道、天气、光照和历史事故转成道路风险逻辑。

## 业务问题与评价指标

根据道路类型、天气、光照、速度限制、曲率、车道数和事故历史预测 `accident_risk`，指标为 RMSE。业务上可用于道路安全评估和风险地图。

当前最佳为 `outputs/submission_cat80_lgbm10_xgb10.csv`，Public RMSE `0.05563`，Private RMSE `0.05588`，OOF RMSE `0.056087`，约前 46%。

## 我的整体解决思路

我把这个项目拆成五步：

1. **先理解数据**：看目标分布、字段类型、缺失模式、train/test 差异和指标特点。
2. **再做清洗**：把明显的脏值、哨兵值、语义缺失和泄露风险处理干净。
3. **重点做特征工程**：不是机械造列，而是把业务直觉、数据结构和模型能力连接起来。
4. **用 OOF 做验证**：每个模型都尽量通过交叉验证产生 OOF，避免只看一次线上提交。
5. **最后复盘取舍**：线上不涨的复杂方案不保留，最终只保留稳定、能解释、能复现的版本。

## EDA 与数据理解

- 事故风险显然不是单变量问题，速度、曲率、天气、光照会相互放大。
- 类别场景非常重要，例如同样速度在不同 road_type 下风险不同。
- 外部原始数据曾测试，但由于分布偏移导致 OOF 变差，最终不使用。

## 数据清洗策略

- 检查类别字段并保留 CatBoost 原生类别处理能力。
- 对除法比例特征加保护，避免除零和异常值。
- 对外部数据做 OOF 验证，发现负收益后主动放弃，避免引入分布噪声。

## 特征工程：不是堆字段，而是写入业务逻辑

| 特征设计 | 为什么这样构造 | 预期收益 |
| --- | --- | --- |
| speed_x_curvature | 高速和急弯同时出现时风险会叠加。 | 模拟车辆动力学风险。 |
| speed_per_lane | 同样速度下车道少可能拥挤和容错更低。 | 表达道路承载压力。 |
| accidents_x_speed | 历史事故多的路段在高速条件下更危险。 | 融合历史风险和当前条件。 |
| weather_light_risk | 恶劣天气和低光照会共同降低能见度。 | 捕捉场景叠加风险。 |
| road/weather/light 交叉 | 道路风险强依赖场景组合。 | 帮助模型区分具体道路环境。 |

这部分是整个项目最值得讲的地方。我的处理原则是：**先解释变量为什么可能影响目标，再决定把它做成比例、计数、交叉、分箱、rank、文本向量还是序列语义表示**。这样做出来的特征不是“玄学调参”，而是有业务含义、有验证闭环的建模资产。

## 模型选择：为什么用这些模型

- CatBoost GPU 负责原生类别特征，是当前最强单模型。
- LightGBM/XGBoost 使用 count encoding 和严格 OOF target encoding。
- 最终融合 80% CatBoost + 10% LightGBM + 10% XGBoost。

我没有把模型当成黑箱堆叠，而是根据数据形态选择模型：结构化数据优先树模型，类别特征多时重视 CatBoost，高维匿名数值用 LightGBM，文本任务则先用 TF-IDF 建强 baseline，再用 Transformer 补语义理解。

## 验证与防止过拟合

- 使用 OOF 或交叉验证观察本地泛化表现。
- 区分 public/private 分数，避免过度追逐单次 public leaderboard。
- 严格排除 ID、target、后竞赛标签等泄露来源。
- 对融合方案做线上复盘：如果复杂方案不涨分，就回退到更稳定版本。

## 运行结果分析

- 当前线上最强来自 CatBoost，说明类别场景组合确实是核心。
- OOF 权重搜索本地略优但线上不涨，说明验证和 leaderboard 存在一定差异。
- 项目未达 top 20%，下一步应该研究数据生成公式，而不是继续盲目堆模型。

- **模型选择能力**：知道什么时候用 CatBoost、LightGBM、XGBoost、线性模型或 Transformer。
- **实验复盘能力**：能解释为什么某个方案涨分，为什么某个方案被放弃。

## 如何复现

安装依赖：

```bash
pip install -r requirements.txt
```

复现时先从 Kaggle 下载原始数据到项目约定的数据目录。部分仓库为了保持轻量，只保留最佳提交文件、实验日志和核心说明；如果仓库中存在 `src/`、`notebooks/` 或 `kaggle_kernel_*`，优先从这些入口运行训练。

常见入口示例：

```bash
python src/train_best.py
# 或在 Kaggle 上运行 kaggle_kernel_* 中的 GPU kernel
```

如果当前项目只保留了最佳产物，可直接查看 `outputs/` 中的 OOF、prediction、submission 和实验摘要文件。
